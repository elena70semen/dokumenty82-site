## What Counts As A Release Artifact

A release artifact is the immutable static export for one site revision:

- generated `out/` directory;
- matching `release-proof/artifact-manifest.json`;
- commit SHA from the build;
- checksum set for key files and routes;
- deployment timestamp or release ID.

For CI builds, the source of truth should be:

- `dokumenty82-static-export-${commitSha}`;
- `dokumenty82-release-proof-${commitSha}`.

For manual emergency deployment, the artifact must still be built from a clean repository checkout and accompanied by an artifact manifest generated by:

```bash
npm run build
npm run release:manifest
```

## Where Previous Artifacts Live

Previous artifacts should be retained as sibling release directories under the server release history. A rollback must point production back to a previous known-good artifact, not rebuild unknown code directly on the server.

Minimum required previous-artifact data:

- release ID;
- source commit SHA, if known;
- build origin: CI artifact or manual emergency artifact;
- manifest checksum file, if available;
- validation status from the previous production verification.

## Rollback Procedure

Run the operational steps inside a persistent terminal session such as `tmux` so that an SSH disconnect does not interrupt the rollback.

High-level rollback steps:

1. Identify the current release ID.
2. Identify the previous known-good release ID.
3. Confirm the previous release directory exists and contains static HTML, `robots.txt` and `sitemap.xml`.
4. Switch the production `current` pointer to the previous release.
5. Validate the web server configuration.
6. Reload the web server.
7. Verify production URLs and metadata.
8. Record the rollback evidence.

Do not paste real SSH connection strings, private key paths, passwords or provider panel credentials into this runbook.

## Robots / Sitemap / Canonical Checks

After rollback, verify:

- `robots.txt` returns `200`;
- `robots.txt` does not accidentally block `/` unless the rollback is intentionally returning to technical HOLD;
- `robots.txt` points to `https://dokumenty82.ru/sitemap.xml`;
- `sitemap.xml` returns `200`;
- sitemap URLs use only `https://dokumenty82.ru`;
- public pages have canonical URLs on `https://dokumenty82.ru`;
- held routes remain excluded/noindex;
- `/internal/` remains blocked.

## If Production Shows A Technical Stub

1. Treat the state as `HOLD_CONFIRMED_TECHNICAL_LAUNCH`.
2. Do not make SEO/content changes first.
3. Verify whether `current` points to the intended release.
4. Check whether the static artifact contains real site HTML or placeholder HTML.
5. Restore the previous known-good public artifact if available.
6. Record the checked URL, status, title and first-page content marker.

## If Production Shows An Old Artifact

1. Compare production title/canonical/H1 with the expected release proof.
2. Check whether the `current` pointer references an older release.
3. Check whether web server cache/CDN/provider cache is serving stale files.
4. Restore or redeploy only a known artifact.
5. Re-run the required production URL checks.

## Rollback Approval

Rollback should be confirmed by:

- owner or delegated project lead;
- operations/deploy responsible person;
- QA/release checker after URL verification.

Emergency rollback may be performed first to restore availability, but evidence must be recorded immediately after.

## Data Forbidden In Rollback Evidence

Never write:

- passwords;
- SSH private keys;
- provider credentials;
- panel screenshots with secrets;
- API tokens;
- webhooks/endpoints with secrets;
- `.env` content;
- client documents;
- personal data from forms/messages.
